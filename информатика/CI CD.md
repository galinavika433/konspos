**CI/CD** — это подход к разработке ПО, который автоматизирует этапы интеграции, доставки и развертывания кода. Главная цель: быстрее выпускать надежные обновления, снижая ручной труд и количество ошибок.


| Термин                          | Значение                  | Суть                                                                                                                           |
| ------------------------------- | ------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **CI** (Continuous Integration) | Непрерывная интеграция    | Разработчики часто (несколько раз в день) сливают код в общую ветку. Каждый слив проверяется автоматической сборкой и тестами. |
| **CD** (Continuous Delivery)    | Непрерывная доставка      | Код всегда готов к релизу после CI. Развертывание на production — **ручное**, но нажатием одной кнопки.                        |
| **CD** (Continuous Deployment)  | Непрерывное развертывание | Следующий шаг: каждое успешное изменение автоматически уходит в production. Без участия человека.                              |

## Основные этапы пайплайна CI/CD

1. **Фиксация кода** → разработчик пушит изменения в Git.

2. **Запуск CI** → автоматически срабатывает триггер (webhook).

3. **Сборка** → компиляция, установка зависимостей.

4. **Тестирование** → модульные, интеграционные, линтеры, проверка безопасности (SAST).

5. **Публикация артефакта** → сохранение собранного образа/бинарника в реестр (Docker Hub, Nexus, Artifactory).

6. **Доставка (CD)** → автоматическое развертывание на staging / предрелизное окружение.

7. **Ручной или автоматический релиз** → запуск в production.



## Главные преимущества

- **Раннее обнаружение багов** — тесты прогоняются после каждого коммита. 
- **Снижение рисков релиза** — маленькие изменения проще откатить.  
- **Автоматизация рутины** — не нужно вручную собирать, тестировать, настраивать окружение.  
- **Быстрая обратная связь** — разработчик узнает о поломке через 5–15 минут.  
- **Дисциплина в команде** — без работающего пайплайна код не попадет в основную ветку.

## Когда стоит внедрять?

- Команда из 2+ разработчиков.

- Проект живет больше 2 месяцев.

- Есть регрессионные баги при релизах.

- Релизный процесс занимает часы/дни ручной работы.

- Нужно часто выпускать обновления (SaaS, мобильные приложения, микросервисы).
